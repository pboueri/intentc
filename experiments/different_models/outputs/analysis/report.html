Different Models Experiment

Comparing Claude Haiku, Sonnet, and Opus at varying effort levels on a zoo builder game task

Key Findings

Haiku cannot complete this task. It failed on both attempts, unable to pass animal breeding validations after exhausting retries. This establishes a clear capability floor for the zoo builder spec.

All Sonnet and Opus variants scored GOOD. Every successful build produced a playable zoo builder game with enclosures, animals, economy, and game-over mechanics. The quality evaluator (Opus 4.6, high effort) did not differentiate between them at the score level.

Higher effort trends toward more code for Sonnet (1191 → 1283 → 1605 LOC), but Opus shows no clear trend (1584 → 1164 → 1590). Opus medium produced the most concise output with the fewest files.

Bug profiles differ across configurations. Common issues include missing camera panning, no sell/remove mechanic, and purely numerical visitors. But crash bugs (sonnet-high restart crash) and logic bugs (sonnet-high breeding) appear at specific configurations only.

Quality Evaluations

Each build was evaluated by Claude Opus 4.6 (high effort) against the original zoo builder specification.

Haiku — Default FAILED

Build failed on both attempts. Could not pass animal breeding validations after exhausting retries. Enclosures built successfully but animals target failed consistently.

Sonnet — Low GOOD

A well-structured, modular implementation that covers the full core loop: building enclosures, purchasing animals, setting admission prices, breeding, enclosure degradation/repair/upgrade, and a game-over condition. The code is cleanly organized across six files with good separation of concerns and readable logic. The main shortcomings are a missing pause UI, dead code from unused factory functions, truncated species display, and the absence of visual visitor entities — but none prevent the game from being playable and engaging.

  • Dual ID generators risk collisions; factory functions in state.js are dead code
  • No pause button despite full pause/unpause logic implemented
  • Only 5 of 8 species shown due to .slice(0, 5) truncation
  • No biome-compatibility gate on animal placement
  • Visitors are purely numerical — no entities rendered

Sonnet — Medium GOOD

A well-structured and complete implementation that faithfully covers all core spec features: biome-based enclosures on a tile grid, animal purchasing and breeding, happiness/sickness/death mechanics, enclosure degradation and upgrades, a demand-curve-driven economy, and a clean pause/game-over flow. Minor UI limitations and inability to remove misplaced enclosures, but no critical logic bugs.

  • Sidebar has no scrolling — buy-animal buttons can become unreachable
  • No devicePixelRatio handling (blurry on Retina displays)
  • No way to sell/remove animals or enclosures
  • Game-over restart may leave cursor stuck on crosshair
  • Only 2 incompatible species shown in sidebar (hardcoded slice)

Sonnet — High GOOD

Clean module separation with a working core gameplay loop. However, a crash-on-restart bug, a largely non-functional breeding system due to flawed pairing logic, and hardcoded HUD hit regions that break at different resolutions are notable issues that would degrade the player experience.

  • Crash bug: Restart resets economy without required fields, causing undefined.toLocaleString() crash
  • Logic bug: Breeding pairs by sequential array index, silently skipping most valid pairs
  • HUD click detection uses hardcoded pixel coordinates — breaks at different window sizes
  • Camera can pan infinitely past map edge
  • Health reaching 0 has no consequence — death/sickness system is cosmetic

Opus — Low GOOD

Delivers the core gameplay loop with clean modularization and meaningful unit tests. Main gaps are complete absence of camera panning, no capacity enforcement on animal purchases, and visitors being purely mathematical rather than visual entities.

  • No camera panning — parts of 1200x800 map inaccessible on smaller screens
  • No capacity check on animal purchases (overcrowding penalty but no block)
  • Hardcoded constants in helper function create fragile coupling
  • Auto-assigns animals to mismatched biomes without player notification
  • No sell/remove mechanic for enclosures or animals

Opus — Medium GOOD

Clean and readable with good separation of concerns and a basic test suite. Covers all core spec features. Main weaknesses are dead canPlaceAnimal function (no capacity enforcement), disconnected visual vs. economic visitor systems, and no camera panning.

  • canPlaceAnimal is dead code — never called, unconditionally returns true
  • Dead tickAccumulator field in state
  • Admission price updates on blur instead of input
  • Visual visitors completely decoupled from economic visitors
  • No camera panning or scrolling
  • No sell/remove mechanic

Opus — High GOOD

Solid implementation with clean module separation and reasonable test coverage. Covers enclosure placement, animal management, economy, degradation, and upgrades. Main gaps are non-functional camera system, static visitors, and constant-duplication issues.

  • No camera pan/scroll controls despite cameraX/cameraY state fields
  • Space/P pause doesn't preventDefault — scrolls the page
  • Visitors are static dots, not moving entities
  • Upgrade tiers hardcoded inline instead of using imported constant
  • REPAIR_COST_RATIO exported but unused; 0.5 hardcoded instead
  • Hardcoded MAP_WIDTH/HEIGHT in entities.js helper risks divergence
